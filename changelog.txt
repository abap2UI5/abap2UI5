abap2UI5 changelog
=================

Legend
------
* : fixed
! : changed
+ : added

unreleased
----------
* Fixed OpenUI5 1.71 component load: core/FrontendAction hard-depended on
  sap/ui/core/Theming (only present since UI5 1.118), so on 1.71 it 404'd and
  the whole component failed to load. Theming is now resolved lazily (via
  sap.ui.require) only when a THEMING frontend action fires.
* Fixed a noisy 404 on OpenUI5 < 1.118: Component.init warm-loaded
  sap/ui/core/Messaging unconditionally, so on 1.71 the async require 404'd and
  the ui5loader retried loudly via synchronous XHR. The warm-load is now gated
  on the running UI5 version (Lib.hasMessagingModule); older releases keep using
  the sap.ui.getCore().getMessageManager() fallback.
* Fixed "adding element with duplicate id
  'z2ui5DeveloperTools--developerToolsEditor'" in the Developer Tools: the
  dialog was destroyed on close and its fragment re-loaded on the next open,
  which raced the close animation on UI5 1.71 and left the CodeEditor id
  registered. The dialog is now loaded once and reused across open/close, and
  destroyed only when the control itself exits.
* Fixed "failed to load sap/m/footer.js" (404) when opening the Developer
  Tools on OpenUI5 1.71: the dialog footer used a <footer> aggregation, which
  sap.m.Dialog only exposes since ~1.110, so on older releases UI5 tried to
  load a nonexistent sap.m.footer control. Switched to the cross-version
  `buttons` aggregation (since 1.21.1).
* Removed the empty css/style.css entry from manifest.json's
  sap.ui5/resources: it was an empty placeholder stylesheet that UI5 loaded as
  a physical <link>, which 404'd on the ABAP system (the file is served through
  the module preload, not as an ICF resource). No styling is lost.
! The fatal-error popup can no longer be dismissed with Escape: it is now a
  sap.m.Dialog with an escapeHandler that rejects (sap.m.MessageBox always
  closes on Escape and offers no way to suppress it), so a broken app can only
  be left via its explicit Details / Restart actions. Details -> Developer
  Tools -> Escape still returns to the error popup.
! Emphasized the Developer Tools "Close" button so it stands out among the
  footer actions.
* Restored 'unsafe-eval' in the default Content-Security-Policy: the OpenUI5
  1.71 ui5loader evaluates module source as a string, so without it the 1.71
  bootstrap failed with a CSP EvalError ("Failed to load component"). Modern
  UI5 does not use eval; 'unsafe-inline' was already allowed, so the delta is
  marginal. Apps pinning a modern UI5 can drop it via their user exit.
* Fixed user-exit discovery: the interface-implementation lookup in
  z2ui5_cl_a2ui5_context passed a WITH EMPTY KEY table to the classic function
  module SEO_INTERFACE_IMPLEM_GET_ALL, whose IMPORTING parameter is WITH DEFAULT
  KEY. The incompatible table type made the CALL FUNCTION fail, so no
  z2ui5_if_exit implementers were returned and the user exit was silently never
  called. Restored WITH DEFAULT KEY.
! Framework-internal ASSERT guards (src/01) now raise a catchable
  z2ui5_cx_a2ui5_error instead of the uncatchable ASSERTION_FAILED short
  dump, so an unexpected internal state surfaces as a graceful error
  (the error popup / Developer Tools) rather than an ST22 dump. Vendored
  (src/00) and obsolete (src/99) code is unchanged.
+ Added the whitelisted frontend control calls as follow_up_action events: cs_event-control_by_id
  (call a whitelisted method on a control resolved by id), cs_event-control_global (call a
  whitelisted method on a global object - MessageToast, MessageBox, BusyIndicator, Theming) and
  cs_event-binding_call (declarative filter/sort on an aggregation binding). All three are scheduled
  via follow_up_action (or roundtrip-free via _event_client); the interim wrapper methods
  control_call, control_call_by_id and binding_call_by_id existed only on this branch and are gone.
+ Added the message> model on every view slot with automatic validation-message collection
+ Added the demo kit formatter pack to model/formatter (weightStateByValue, stockStatus*, round2DP, dimensions, deliveryStatusState)
! Event arg serialization no longer drops empty middle args: get_t_arg keeps an empty argument
  between filled ones as an '' placeholder (only trailing empties are trimmed), so a CONTROL_BY_ID
  action without a view no longer loses its method position. Apps that passed a possibly-empty MIDDLE arg via
  _event/_event_client/follow_up_action and read the following value by its previously-compacted position
  now receive '' at that slot - pass the value at the intended position or drop the empty arg.
